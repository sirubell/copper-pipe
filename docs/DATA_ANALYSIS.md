# 助教資料集分析

對 `train/OK` (90 張) 與 `train/NG` (10 張) 全部檔名與圖像做的歸納。

## 1. TL;DR — 一句話結論

**OK 不是真實照片，是 InstructPix2Pix 從同一張原圖生成的合成資料；NG 才是真實瑕疵。**
資料集是 **合成 vs 真實** 的不平衡，不只是「正常 vs 瑕疵」的不平衡。

| 類別 | 張數 | 來源 | 標記 |
|---|---|---|---|
| OK (正常) | 90 (100%) | InstructPix2Pix 合成 | 檔名含 `[instructP2P-crop_multi_scale+each_5000]` |
| NG (瑕疵) | 10 (100%) | 真實照片裁切 | 檔名乾淨，沒有合成標記 |

## 2. 檔名格式拆解

完整檔名範例（OK）：
```
Image__2024-11-29__11-25-29_crop_1469_1508_1656_1767_512_1306_1381_1818_1893_r00_20260506103705_[instructP2P-crop_multi_scale+each_5000]_1741216934.png
```

拆成 8 段：

| # | 內容 | 意義 |
|---|---|---|
| 1 | `Image__2024-11-29__11-25-29` | 來源照片拍攝時間（**全資料集都是這一張**，2024-11-29 11:25:29） |
| 2 | `crop_1469_1508_1656_1767` | 原圖上的銅管 bounding box (x1, y1, x2, y2) |
| 3 | `512_1306_1381_1818_1893` | 把 bbox 外擴到 512×512 後的座標（InstructPix2Pix 輸入要 512×512） |
| 4 | `r00` … `r12` | 旋轉增強的索引 (0 ~ 12 共最多 13 個角度) |
| 5 | `20260506103705` | InstructPix2Pix 處理時間戳（2026-05-06 10:37:05） |
| 6 | `[instructP2P-crop_multi_scale+each_5000]` | **合成方法標籤** — 詳見下節 |
| 7 | `1741216934` | Unix timestamp 1741216934 = 2025-03-05 23:22 UTC（推測是 seed 或 batch id） |
| 8 | `.png` | 副檔名 |

NG 檔名只有段 1–4：
```
Image__2024-11-29__11-25-29_crop_1669_1844_1859_2107_512_1508_1719_2020_2231_r02.png
```
也就是「真實照片 → 裁切 → 旋轉」沒有經過任何 AI 合成。

## 3. InstructPix2Pix 是什麼

[InstructPix2Pix (Brooks et al., CVPR 2023)](https://www.timothybrooks.com/instruct-pix2pix/)
是一個 diffusion-based 影像編輯模型 — 輸入一張圖 + 一句文字指令（例如「把這條管子換成乾淨的」），輸出是依指令改寫過的圖。

`[instructP2P-crop_multi_scale+each_5000]` 這個 tag 推測語意：

- `instructP2P` = 用 InstructPix2Pix 模型
- `crop_multi_scale` = 在不同 scale 下裁切再餵入
- `each_5000` = 每個 source crop 跑 5000 步（diffusion steps）或產 5000 張變體

**為什麼助教要這樣做？**
原始 OK 樣本應該不多（推測就是這一張原圖切出的 9 個 crop 區域），用 InstructPix2Pix
合成大量「乾淨銅管」變體，是常見的 **小樣本資料增強** 手法 — 比起傳統旋轉/翻轉/色彩抖動，
diffusion 能改變反光、表面紋理、背景細節，讓 normal 分佈看起來更「豐富」。

### 4.1 來源照片

### 4.4 影像尺寸
- OK：尺寸隨 crop+旋轉變化，例 187×259, 194×264, 198×268（高度寬度都會浮動）

NG 是直接從原圖裁切，所以一致；OK 經過 InstructPix2Pix 的 512×512 → 裁回流程，會有些
微差異。

## 5. 視覺比對

我看了同 crop 區域的 OK vs NG 對：

- **Crop 1669_1844_1859_2107**：OK 表面均勻、反光乾淨；NG 表面有淡淡的不規則紋理與小斑點
- **Crop 2058_1849_2246_2109**：OK 表面乾淨；NG 中央有一塊明顯的淺色斑（典型表面瑕疵）
- **Crop 2254/2452**：類似 — OK 看起來像「修圖過的」乾淨銅管，NG 是真的有缺陷的銅管

整體印象：
1. **OK 太乾淨**：reflection、shading、邊界都很「平滑」，有點「太完美」的 AI 味
2. **NG 有真實感**：刮痕、表面斑塊、邊緣磨損都符合產線實況
3. **人眼分得開**：所以模型也應該分得開（事實上我們的 PatchCore AUROC=0.99 確實做到了）

## 6. 對模型訓練的影響

### 6.1 為什麼會有 False Positive
我們 PatchCore 跑出來的 1 張 FP、EfficientAD 跑出來的 2 張 FP，全部都帶
`[instructP2P-...]` 標記（廢話，OK 全是）。但它們的特徵是：

- **旋轉索引偏極端**：r07、r12 等（InstructPix2Pix 為了補資料硬生出來的）
- **合成痕跡較明顯**：可能是反光被洗掉、表面紋理被過度平滑或產生輕微 hallucination

模型看到這些「合成感比較重」的 OK，分數就會偏高。換句話說：
**模型抓的不完全是「銅管瑕疵 vs 正常」，而是「典型 InstructPix2Pix 輸出 vs 偏離分佈的影像」**。
真實瑕疵分數高是因為「不像 InstructPix2Pix 輸出」；某些合成失敗的 OK 分數高也是同一個理由。

### 6.2 為什麼這樣的資料還是能跑出 AUROC 0.99
因為大部分 InstructPix2Pix 輸出都很穩定一致，而真實 NG 的紋理本來就跟合成 OK 不同。
模型抓的「分佈差異」剛好跟「正常 vs 瑕疵」高度重合，不過不是 100% 一致。

### 6.3 隱含的測試集問題
我們現在的測試集（split 後的 `test/good` 20 張）也全是 InstructPix2Pix 合成的。
這代表：
- 評估指標衡量的是「對 InstructPix2Pix 輸出的記憶能力」+「對真實瑕疵的偵測能力」混合體
- 如果未來把模型部到產線上、餵真實 OK 照片，**分數會跟現在不一樣**（可能變好也可能變差，
  取決於合成跟真實的 domain gap 偏向哪邊）

## 7. 接下來建議

### 7.1 跟助教確認的事
1. OK 全合成是有意的嗎？正式評估時是否也會用 InstructPix2Pix 合成的 OK，還是改用真實照片？
2. 是否能拿到「原始的、未經 InstructPix2Pix 的真實 OK 照片」幾張當 sanity check？
3. 旋轉 r10–r12 在 NG 為何缺失？是真實資料就沒有，還是 NG 沒做這幾個角度？

### 7.2 不需要外部資訊就能做的事
1. **Domain-aware 切分**：現在 train/test 都是 InstructPix2Pix 合成的 OK，可以保留幾張
   檔名 hash 上「相近的合成 seed」當 held-out，看模型對 InstructPix2Pix 多樣性的泛化能力。
2. **濾掉極端旋轉**：r10–r12 是合成擴增、NG 沒對應，丟掉這 23 張看看 AUROC 變化（如果分數
   反而升高，代表這些圖是雜訊；如果反而降低，代表它們提供了有用的多樣性）。
3. **單獨評估「真實感」**：訓練一個小的 binary classifier（real vs synthetic），看 OK
   pool 裡 InstructPix2Pix 的 detectability。FP 應該都是 detectability 比較低的合成圖。
4. **跑 ensemble + TTA**：之前提過的，目前看起來這個 dataset 的瓶頸是合成圖品質而不是模型，
   ensemble 跟 TTA 應該能把 0.99 → 1.0 推上去。

### 7.3 建議報告裡明講的事
如果這份作業要交報告，我會在 limitations 段加上：
> 評估集中的「正常」樣本全為 InstructPix2Pix 合成；本實驗實際上量測的是模型對
> 「合成正常 vs 真實瑕疵」的區分能力，而非「真實正常 vs 真實瑕疵」。實際部署
> 表現可能因 domain gap 而異。

這比硬把分數催到 1.0 更誠實，也比較好看。
